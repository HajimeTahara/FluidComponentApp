# 複数流体系統ネットワーク設計方針

Date: 2026-06-18
Status: policy implemented

## 目的

定常流れ解析を、単一の流体物性を持つ1つの配管ネットワークから、複数の独立した流体系統を同一ダイアグラム上で扱える構成へ拡張する。

最初の対象は、次の設計課題への対応とする。

- ネットワーク全体で1つだけ保持している `density` / `viscosity` を、流体系統ごとの物性へ分離する。
- ダイアグラム上に、水ループ、空気ループなどの流体系統を識別する概念を追加する。
- バックエンドの部品計算ロジックを、部品種類ごとの責務へ分離し、熱交換器やタービンなどの連成要素を追加しやすくする。

二流体熱交換器、温度依存物性、タービンとポンプの動力連成は次段階で扱う。今回の方針では、その前提となるデータ構造と計算責務の整理を優先する。

## 現状の問題

### 1. 物性がネットワーク全体で1つ

現状の `PipeNetworkRequest` は、リクエスト全体に `density` と `viscosity` を1つだけ持つ。フロントエンドもツールバー上の単一入力として `density` / `viscosity` を保持している。

この構造では、同一ダイアグラム上に水ループと空気ループを配置しても、すべてのパイプ、バルブ、ポンプ、熱交換器が同じ物性で計算される。

### 2. 流体系統の概念がない

現状のノードとエッジには、どの流体ループに属するかを示す `fluidSystemId` がない。複数の閉ループを描画することはできるが、計算上は1つの圧力ネットワークとして扱われる。

そのため、次の区別ができない。

- 水配管と空気ダクト
- 同じ圧力節点として接続されるべきラインと、熱的にだけ結合するライン
- 流体接続と、将来追加する軸動力/電力接続

### 3. 部品計算が条件分岐に散らばっている

バックエンドでは、パイプ、ポンプ、熱交換器、拡縮管、エルボ、バルブの処理が複数の `if/elif` に分散している。

このまま部品を追加すると、次の処理すべてに同じ種類判定を増やす必要がある。

- 圧力差から流量を逆算する処理
- 流量から圧損/昇圧を計算する処理
- 初期圧力推定
- 結果生成
- 温度伝播
- フロントエンドの結果表示と特性表示

二流体熱交換器やタービンのように、水力・熱・動力をまたぐ部品では、この構造のままでは変更範囲が大きくなりすぎる。

## 基本方針

### 流体系統を一級データとして追加する

解析リクエストに `fluidSystems` を追加する。

```text
fluidSystems: [
  {
    id: "water-loop",
    name: "水ループ",
    fluid: "Water",
    propertyMode: "constant",
    density: 1000.0,
    viscosity: 0.001,
    specificHeat: 4184.0
  },
  {
    id: "air-loop",
    name: "空気ループ",
    fluid: "Air",
    propertyMode: "constant",
    density: 1.2,
    viscosity: 0.000018,
    specificHeat: 1005.0
  }
]
```

初期実装では `propertyMode: "constant"` を基本とし、CoolPropによる温度・圧力依存物性は後段で追加する。

### ノードまたはエッジに所属流体系統を持たせる

2ポート流体要素は `fluidSystemId` を持つ。パイプ、ポンプ、熱交換器の片側流路、拡縮管、エルボ、バルブは、所属する流体系統の物性を使って計算する。

```text
node.params.fluidSystemId = "water-loop"
```

境界ノードにも `fluidSystemId` を持たせ、どの流体系統へ圧力/流量/温度条件を与えるかを明示する。

当面はノード単位で `fluidSystemId` を持つ方式を採用する。エッジ単位の所属は、将来「配管をノードではなくエッジとして扱う」設計に移行する場合に再検討する。

### 系統ごとに圧力ネットワークを分割して解く

バックエンドでは、受け取ったノードとエッジを `fluidSystemId` ごとのサブグラフに分割する。各サブグラフを独立した圧力ネットワークとして解く。

同じダイアグラム上に複数ループが存在しても、水ループと空気ループの節点圧は互いに直接連結しない。

連成が必要な部品は、通常の流体ラインとは別のカップリング要素として扱う。

## 部品モデル分離方針

バックエンドに部品モデルの薄い抽象化を導入する。

各部品モデルは、少なくとも次の責務を持つ。

- `flow_from_dp(params, dp, fluid)`: 圧力差から流量を返す。
- `result_from_flow(params, q, fluid)`: 流量から圧損/昇圧と表示用結果を返す。
- `initial_pressure_delta(params, q, fluid)`: 初期圧力推定用の圧力差を返す。
- `thermal_update(params, q, fluid, thermal_state)`: 温度変化を返す。温度変化がない部品は入口温度をそのまま返す。

既存の関数群は、このモデル層へ段階的に移す。

初期移行対象は次の通り。

- pipe
- pump
- heatExchanger
- reducer
- elbow
- valve

`_solve_boundary_network` は部品種類ごとの詳細式を直接知らず、モデルの共通インターフェースを呼ぶだけにする。

## フロントエンド方針

### 流体系統パネルを追加する

定常流れ解析画面に、流体系統を管理する小さなパネルを追加する。

初期機能は次とする。

- 系統名
- 流体名
- 密度
- 粘度
- 比熱
- 表示色

初期状態では `水ループ` を1つ作成し、既存挙動と互換にする。

### 部品パラメータに流体系統選択を追加する

境界、パイプ、ポンプ、熱交換器、拡縮管、エルボ、バルブの設定パネルに、所属流体系統を選ぶUIを追加する。

初期値は最初の流体系統とする。既存の保存データや古いノードに `fluidSystemId` がない場合も、最初の流体系統へフォールバックする。

### グローバル物性入力は系統管理へ移す

現状のツールバーにある単一の `密度 ρ` / `粘度 μ` 入力は、互換期間を経て流体系統パネルへ移す。

移行初期は、最初の流体系統の密度・粘度として扱い、既存UIの操作感を壊さない。

## API移行方針

初期移行では後方互換を保つ。

- `fluidSystems` がある場合は、系統ごとの物性を使う。
- `fluidSystems` がない場合は、従来通り `density` / `viscosity` を使って単一系統として扱う。
- `fluidSystemId` がないノードは、デフォルト系統へ割り当てる。

これにより、既存の単一ループ解析を壊さずに複数流体系統へ移行する。

## 次段階で扱うこと

### 二流体熱交換器

現在の熱交換器は、流体が固定温度 `exchangeTemperature` に向かって熱交換する単一流路モデルである。複数流体系統対応後に、次のような二流体カップリング部品へ拡張する。

- hot側入口/出口ポート
- cold側入口/出口ポート
- UA
- 両側の質量流量と比熱
- 熱交換量
- hot側/cold側出口温度

### タービンと動力連成

タービンは流体から動力を取り出す部品として扱う。ポンプへ動力を供給するには、流体ラインとは別に `powerConnection` または `shaftBus` を追加する。

初期方針では、流体ネットワークと動力ネットワークを分離し、あとから連成計算できるようにする。

## 実装順序

1. `fluidSystems` と `fluidSystemId` を型とAPIに追加する。
2. フロントに流体系統パネルと部品ごとの系統選択を追加する。
3. バックエンドでサブグラフを流体系統ごとに分割して解く。
4. 部品計算を共通インターフェースへ段階的に移す。
5. 既存の単一物性入力をデフォルト流体系統へ移行する。
6. 二流体熱交換器の設計に進む。
7. タービン/動力連成の設計に進む。

## 完了条件

- 同一ダイアグラム上に水ループと空気ループを作成できる。
- 各ループが異なる密度・粘度・比熱を持てる。
- 各ループの圧力分布と流量が、互いに混線せず個別に解かれる。
- 既存の単一ループ解析が後方互換で動く。
- 新しい部品を追加するとき、圧力ソルバー本体への変更が最小になる。
